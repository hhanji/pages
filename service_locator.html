Service Locator -suunnittelumalli Pythonissa

Mikä on Service Locator?

Service Locator on malli, jossa luokka tai komponentti voi hakea tarvitsemansa palvelun keskitetystä rekisteristä eli “palvelulokaattorista”. Tämä rekisteri tuntee kaikki palvelut ja palauttaa niitä pyynnöstä.

Tarkoitus: Tarjota keskitetty paikka, josta riippuvuuksia (palveluita) voidaan hakea ilman, että niitä injektoidaan manuaalisesti.

Milloin käyttää?

Service Locator -mallia voidaan käyttää kun:

  • Halutaan hallita monia palveluita keskitetysti.
  • Tarvitaan globaali pääsy tiettyihin palveluihin.
  • Dependency Injection on liian raskas tai monimutkainen ratkaisu.

Varoitus: Haittoja

❗ Service Locator piilottaa riippuvuudet, mikä vaikeuttaa testattavuutta ja tekee koodista tiukemmin kytkettyä.

Jos tarvitset hyvin testattavaa ja löyhästi kytkettyä arkkitehtuuria, Dependency Injection on yleensä parempi valinta.

Esimerkki: Python Service Locator

1. Palveluiden määrittely

class EmailService:
    def send(self, message):
        print(f"Lähetetään sähköposti: {message}")

class SMSService:
    def send(self, message):
        print(f"Lähetetään tekstiviesti: {message}")

2. Service Locator -luokka

class ServiceLocator:
    _services = {}

    @classmethod
    def register(cls, name: str, service):
        cls._services[name] = service

    @classmethod
    def get(cls, name: str):
        service = cls._services.get(name)
        if service is None:
            raise Exception(f"Palvelua '{name}' ei löydy.")
        return service

3. Palveluiden rekisteröinti ja käyttö

# Rekisteröidään palvelut
ServiceLocator.register("email", EmailService())
ServiceLocator.register("sms", SMSService())

# Käytetään palveluita
email = ServiceLocator.get("email")
email.send("Tervetuloa!")

sms = ServiceLocator.get("sms")
sms.send("Koodisi on 12345")

Tulostus:

Lähetetään sähköposti: Tervetuloa!
Lähetetään tekstiviesti: Koodisi on 12345

Testattavuusongelma

# Jos NotificationManager hakee palvelun lokaattorista:
class NotificationManager:
    def notify(self, message):
        service = ServiceLocator.get("email")
        service.send(message)

🔴 Nyt NotificationManager on tiukasti sidottu lokaattoriin eikä ole helposti testattavissa ilman monimutkaista mockausta.

Ratkaisu: Yhdistetty malli

Joskus voidaan yhdistää Service Locator ja Dependency Injection:

class NotificationManager:
    def __init__(self, service=None):
        self.service = service or ServiceLocator.get("email")

    def notify(self, message):
        self.service.send(message)

✅ Tällöin testissä voidaan antaa mock_service suoraan:

class MockService:
    def send(self, message):
        print(f"[MOCK] {message}")

mock = MockService()
notifier = NotificationManager(mock)
notifier.notify("Testiviesti")

Yhteenveto: Edut ja haitat

Hyödyt Haitat
Keskitetty palvelujen hallinta Piilottaa riippuvuudet
Yksinkertainen toteuttaa Heikentää testattavuutta
Vähentää tarvetta manuaaliselle injektiolle Tiukempi kytkentä luokkien välillä
Soveltuu pieniin tai väliaikaisiin ratkaisuihin Voi johtaa huonoon arkkitehtuuriin isossa sovelluksessa

Käyttökohteita

  • Pienet tai keskisuuret sovellukset
  • Pelit (esim. globaali resurssihallinta)
  • Skriptit ja työkalut, joissa ei haluta monimutkaista DI-rakennetta
  • Nopean prototyypin rakentaminen

Service Locator voi olla hyödyllinen pienissä sovelluksissa tai kun halutaan nopeasti yhdistää palveluita ilman erillistä riippuvuuksien injektiomekanismia. Kuitenkin isommissa järjestelmissä Dependency Injection on usein selkeämpi ja testattavampi ratkaisu.
