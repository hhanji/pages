Twin-suunnittelumalli Pythonissa

Mikä on Twin-malli?

Twin-suunnittelumalli erottaa yhden olion kahdeksi toisiinsa liittyväksi olioksi:

  • Looginen olio (logic) – sisältää liiketoimintasäännöt ja toiminnallisuuden
  • Fyysinen olio (representation) – vastaa olion fyysisestä käyttäytymisestä, kuten sijainnista, animaatiosta tai tilasta

Tarkoitus: Tarjota vaihtoehto perinteiselle periytymiselle silloin, kun halutaan joustava tapa yhdistää kahden eri näkökulman vastuut yhdelle kokonaisuudelle ilman syvää perintähierarkiaa.

Missä Twin-mallia käytetään?

Sovelluskohde Esimerkki
Pelit Pelihahmon looginen tila vs. animaatio
Käyttöliittymät UI-komponentin logiikka vs. ulkoasu
Simulaatiot Fysiikkalogiikka vs. visuaalinen esitys
Robotit Toimintalogiikka vs. liikkeen hallinta

Python-esimerkki: Pelihahmo ja sen grafiikka

1. Looginen olio – pelihahmon tila ja toiminta

class PlayerLogic:
    def __init__(self):
        self.health = 100
        self.position = (0, 0)
        self.twin = None  # viittaus grafiikkaan

    def move(self, x, y):
        self.position = (x, y)
        print(f"[LOGIIKKA] Hahmo liikkuu kohtaan {self.position}")
        if self.twin:
            self.twin.update_position(x, y)

    def take_damage(self, amount):
        self.health -= amount
        print(f"[LOGIIKKA] Hahmo ottaa {amount} vahinkoa, hp = {self.health}")
        if self.twin:
            self.twin.flash_red()

2. Fyysinen olio – hahmon visuaalinen esitys

class PlayerGraphics:
    def __init__(self):
        self.position = (0, 0)
        self.twin = None  # viittaus logiikkaan

    def update_position(self, x, y):
        self.position = (x, y)
        print(f"[GRAFIIKKA] Päivitetään animaatio kohtaan {self.position}")

    def flash_red(self):
        print("[GRAFIIKKA] Hahmo välähtää punaisena (vahinko!)")

3. Twinien yhdistäminen

# Luodaan instanssit
logic = PlayerLogic()
graphics = PlayerGraphics()

# Kytketään twin-viitteet
logic.twin = graphics
graphics.twin = logic

4. Käyttö

logic.move(5, 10)
logic.take_damage(20)

Tulostus:

[LOGIIKKA] Hahmo liikkuu kohtaan (5, 10)
[GRAFIIKKA] Päivitetään animaatio kohtaan (5, 10)
[LOGIIKKA] Hahmo ottaa 20 vahinkoa, hp = 80
[GRAFIIKKA] Hahmo välähtää punaisena (vahinko!)

Hyödyt

✅ Selkeä jako loogiseen ja visuaaliseen/tekniseen vastuuseen
✅ Ei tarvita moniperintää tai sekavia luokkahierarkioita
✅ Helppo testattavuus (logiikka ilman grafiikkaa)
✅ Eri twin-komponentteja voidaan vaihtaa (esim. eri näkymä käyttöliittymälle)

Haitat

⚠️ Twin-viittaukset on hallittava manuaalisesti
⚠️ Ei ole sisäänrakennettu malli Pythonissa → täytyy toteuttaa itse
⚠️ Mahdolliset sykliset riippuvuudet, jos ei olla tarkkana

Twin-malli vs MVC

Malli Vastuujako
Twin Kaksi oliota hoitaa eri näkökulmia yhdestä entiteetistä (esim. logiikka & ulkoasu)
MVC Malli, näkymä ja kontrolleri ovat erilliset ja yleisempi rakenne monille näkymille

Twin-suunnittelumalli tarjoaa hyvän rakenteen tilanteisiin, joissa yksi olio tarvitsee kaksi erillistä roolia – esimerkiksi logiikkaa ja esitystä. Pythonissa se on helppo toteuttaa pitämällä kaksi oliota tiiviisti linkitettynä toisiinsa.
